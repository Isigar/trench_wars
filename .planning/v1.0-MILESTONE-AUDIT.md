# v1.0 Milestone — Cross-Phase Integration Audit

**Auditor:** Cross-phase integration verifier (Opus 4.7 1M)
**Date:** 2026-05-15
**Scope:** All 9 phases (Foundations → Polish), 120 plans, 1303 Pest + 139 bot Vitest + 40 rcon-worker Vitest tests.
**Verdict:** **gaps_found** — automated gates GREEN at every phase boundary, but a load-bearing **web → bot dispatcher gap** silently breaks 3 of the 8 outbound message kinds at runtime. The web outbox correctly enqueues `article_announce` (Phase 7), `match_result_announce` (Phase 8) and `user_dm` (Phase 9) rows; the bot's render dispatcher throws `Unknown message_type` on every one of them, so the Discord side-effect is never produced. This is not visible in tests because Phase 7/8/9 tests assert outbox-row state only; the bot Vitest suite has no end-to-end coverage of the new kinds.

## BLOCKER findings

### BLOCKER 1 — Bot render.ts is missing dispatchers for 3 outbound message_type values

**Location:** `apps/bot/src/services/render.ts:85-107` (the `render(client, row)` switch).

**What's missing:**
- No handler for `message_type='article_announce'` (Phase 7, written by ArticleObserver).
- No handler for `message_type='match_result_announce'` (Phase 8, written by MatchResultObserver).
- No handler for `message_type='user_dm'` (Phase 9, written by `App\Notifications\Channels\DiscordChannel`).

**Symptom at runtime:** The bot polls `/api/bot/outbound-messages?status=pending` (does not filter by message_type), fetches every pending row, calls `render(client, row)`, which throws `Error: [bot/render] Unknown message_type: <kind>`. The worker catches the throw and POSTs `/outbound-messages/{id}/failed` with the error string. The row's `status` flips `pending → failed` and the Filament admin retry surface (`DiscordOutboundMessageResource`) shows it. **No Discord side-effect ever happens** for these 3 kinds.

**Why automated tests don't catch this:**
- Phase 7's `ArticleAnnounceOutboundTest` asserts the outbox row exists with correct payload shape — does not exercise the bot.
- Phase 8's `ScrimE2EHappyPathTest` asserts the same — does not exercise the bot.
- Phase 9's `DiscordChannelOutboxTest` asserts the same plus `Http::assertNothingSent` — does not exercise the bot.
- Bot Vitest suite was last extended in Phase 6 (`tournamentEmbeds.test.ts`); test counts confirm Phase 7/8/9 added ZERO bot tests (`bot_test_count: 139` unchanged across the final 3 phases).
- The web→bot seam is tested end-to-end only for `match_announce` and `role_sync` (the original Phase 5 kinds) plus the three tournament kinds added in Phase 6.

**Side note:** Even for the tournament kinds that ARE in render.ts, the `renderTournamentAnnounce` and `renderBracketResultAnnounce` paths do `client.channels.fetch(row.channel_id)` where `channel_id=''` (TournamentObserver writes empty string at D-06-10-E; BracketAdvancementService does the same per D-06-08-A continuation). The render.ts docblock at line 184-186 acknowledges the gap ("the bot worker resolves the channel via env.DISCORD_ANNOUNCE_CHANNEL_ID OR the organising clan's announce channel") but the implementation never reads `env.DISCORD_ANNOUNCE_CHANNEL_ID` or any other fallback before calling `client.channels.fetch('')`. **This is a second runtime defect** that compounds the v1.0 ship risk: the tournament_announce and bracket_result_announce paths are also broken once the web side writes the placeholder empty channel_id (which is the production code path — verified by the SC-1 TournamentEndToEndTest setup).

**Required fix (recommended scope for a Phase 10 hotfix or pre-ship patch):**
1. Add to `apps/bot/src/lib/embeds.ts`: `buildArticleAnnounceEmbed`, `buildMatchResultAnnounceEmbed`, `buildUserDmContent`.
2. Add to `apps/bot/src/services/render.ts`: dispatch branches for the 3 new kinds.
3. Add to `apps/bot/src/services/render.ts`: fallback channel resolution when `row.channel_id === ''` — read `env.DISCORD_ANNOUNCE_CHANNEL_ID` (does not exist in `apps/bot/src/env.ts` today — needs adding) or look up the host clan's announce channel via an API call.
4. Extend `apps/bot/src/types/apiContracts.ts` discriminated union to include `article_announce`, `match_result_announce`, `user_dm`.
5. Bot Vitest cases per kind, ideally an end-to-end vitest case that exercises render(client, row) against a fake discord.js Client for each of the 7 kinds.
6. user_dm in particular needs `client.users.fetch(recipient_id)` + DM channel creation, since it targets a USER not a CHANNEL — different code path than channel.send.

## WARNING findings

### WARNING 1 — TournamentObserver/BracketAdvancementService/ArticleObserver write `channel_id=''` placeholder; render.ts doesn't resolve

The placeholder channel_id pattern (D-06-10-E, D-08-12-A precedent) is intentional on the web side, but the bot render path was never updated to resolve it. For TournamentObserver-written rows the docblock claims "resolved via env.DISCORD_ANNOUNCE_CHANNEL_ID OR the organising clan's announce channel" but neither lookup exists in the code. For ArticleObserver the same: empty `channel_id`, no fallback in the bot. This means even the kinds the bot CAN render fail at the channel resolution step in production.

For `match_result_announce` (Phase 8) the channel_id is correctly populated by `MatchResultObserver.php:170-187` (it reads `$match->hostClan?->discord_announce_channel_id`). So Q6's failure mode is purely the missing dispatcher, not the channel — the row arrives with a valid channel id and immediately throws Unknown message_type.

`bot_test_count: 139` in Phase 6 frontmatter. Phase 7, 8, 9 each preserve `bot_test_count: 139`. This is structurally how the BLOCKER above stayed invisible — the bot test surface was frozen at Phase 6's contract. Every phase that introduced a new outbound kind should have shipped a paired bot Vitest test exercising the render.ts dispatch branch.

### WARNING 3 — apiContracts.ts OutboundRow message_type union is stale

`apps/bot/src/types/apiContracts.ts:59-61` declares the message_type union as `'tournament_announce' | 'tournament_announce_update' | 'bracket_result_announce'` (plus `match_announce` and `role_sync` from Phase 5 at adjacent lines). The TypeScript discriminated union does NOT include `article_announce`, `match_result_announce`, or `user_dm`. Combined with the bot tsc-strict gate passing, this means the bot type system actively forbids the bot from receiving these kinds — and the web side is producing them. The contract mismatch is structural.

## Final verdict

**Status: gaps_found**

The 9 phases ship a load-bearing v1.0 with one significant cross-phase wiring defect:

> **The web-side observers correctly enqueue 3 new outbound message kinds across Phases 7/8/9 (`article_announce`, `match_result_announce`, `user_dm`), but the Phase 5 bot dispatcher was never extended to handle them. In production, every one of these rows fails at the bot render step and never produces a Discord side-effect.**

This breaks:
- Phase 7 SC-5 ("Discord announce on publish is wired") — the announce happens at the outbox level only.
- Phase 8's tail-of-chain Discord embed (the operator-expected match result announcement at the end of the SC-5 scrim happy path).
- Phase 9 SC-1's Discord DM channel ("Discord DM rules wired") — DMs never deliver.

The web-side data integrity is intact. Match results, leaderboards, articles, audit logs, FTS, brackets, standings, RCON ingest — all functionally complete and proven by the 1303-test suite.
