<objective>
Wave 4 (parallel to plan 05-05): ship the SC-4 substrate — when a player joins/leaves a clan on the website, a Horizon-retried `SyncDiscordRolesJob` writes a `role_sync` outbound row that plan 05-11's bot worker picks up and converts into a Discord REST `PUT/DELETE /guilds/{g}/members/{u}/roles/{r}` call.

Per RESEARCH Pattern 5, the job does NOT call Discord directly. It writes a `discord_outbound_messages` row with `message_type='role_sync'`. This unifies the v1 delivery model: every Discord side-effect (match announces, role syncs) flows through the SAME outbox table, with the SAME bot worker, and the SAME retry semantics. Horizon's retry is the OUTER layer (dispatch retry if the JOB itself crashes); the outbound table's backoff_until is the INNER layer (delivery retry if Discord 429s or 5xxes).

The ClanMembershipObserver is what links Phase 2's clan membership lifecycle to this new behaviour. Phase 2 plans 02-10/02-11 created the ClanMembership model and its services (ClanInviteService, ClanApplicationService) — they create + update ClanMembership rows. By hooking the model's observer, we capture every relevant lifecycle event without modifying the existing services.

Defensive pre-flight: skip dispatch when user.discord_id or clan.discord_role_id is empty. These should be populated in normal operation (Phase 1 OAuth ensures discord_id; admin sets discord_role_id via Phase 2 plan 02-13 Filament resource); the skip is a guard against bad data + a hook for tests that don't seed these values.

Purpose: SC-4 — "joining/leaving a clan on the website triggers Discord role assign/remove via Horizon-retried jobs."

Output: 1 job + 1 observer + ClanMembership model amendment (booted() registers observer) + 2 GREEN test files.

SyncDiscordRolesJob shape (verbatim per RESEARCH Pattern 5 + Horizon retry verification):

  namespace App\Jobs
  use Illuminate\Bus\Queueable
  use Illuminate\Contracts\Queue\ShouldQueue
  use Illuminate\Foundation\Bus\Dispatchable
  use Illuminate\Queue\InteractsWithQueue
  use Illuminate\Queue\SerializesModels
  use App\Models\ClanMembership
  use App\Models\DiscordOutboundMessage

  final class SyncDiscordRolesJob implements ShouldQueue {
      use Dispatchable, InteractsWithQueue, Queueable, SerializesModels;

      public int $tries = 5;

      public function __construct(
          public readonly string $membershipId,
          public readonly string $action, // 'add' | 'remove'
          public readonly ?string $causerUserId = null,
      ) {}

      /**
       * @return array<int>
       */
      public function backoff(): array {
          return [1, 5, 15, 60, 300]; // seconds; matches outbound table backoff schedule
      }

      public function handle(): void {
          $membership = ClanMembership::with(['user','clan'])->find($this->membershipId);
          if ($membership === null) return; // membership deleted hard; nothing to sync

          $userDiscordId = $membership->user?->discord_id;
          $roleDiscordId = $membership->clan?->discord_role_id;

          if ($userDiscordId === null || $userDiscordId === '' || $roleDiscordId === null || $roleDiscordId === '') {
              // Defensive: skip when binding is incomplete; observer should have caught this.
              return;
          }

          DiscordOutboundMessage::create([
              'channel_id' => '', // role sync has no channel (PUT/DELETE on /guilds/{g}/members/{u}/roles/{r})
              'message_type' => 'role_sync',
              'status' => 'pending',
              'payload' => [
                  'user_discord_id' => $userDiscordId,
                  'role_discord_id' => $roleDiscordId,
                  'action' => $this->action, // 'add' | 'remove'
                  'membership_id' => $membership->id,
                  'clan_id' => $membership->clan_id,
                  'user_id' => $membership->user_id,
              ],
              'causer_user_id' => $this->causerUserId,
          ]);
      }
  }

NOTE on channel_id='' (empty string): the discord_outbound_messages.channel_id column is text NOT NULL (per plan 05-02 migration). For role_sync rows the channel is irrelevant — the bot worker reads message_type='role_sync' and uses the Guilds API instead. Plan 05-02 migration kept the column NOT NULL for match_announce semantics; consider relaxing OR pass empty string per this plan. RECOMMENDATION: keep NOT NULL + pass '' here (defensive — UI cannot accidentally read channel_id for role_sync rows; the JSON payload is the contract).

(Plan 05-02 SHOULD be re-checked — if making channel_id nullable is preferred, this plan ships an amendment migration adding ->nullable(). Recommendation: ship empty string for v1 simplicity; revisit if Filament admin display gets ugly.)

ClanMembershipObserver shape:

  namespace App\Observers
  use App\Models\ClanMembership
  use App\Jobs\SyncDiscordRolesJob
  use Illuminate\Support\Facades\Auth

  final class ClanMembershipObserver {
      public function created(ClanMembership $membership): void {
          if ($membership->left_at !== null) return; // historical seed; not an active join
          $this->dispatchSyncIfBindingComplete($membership, 'add');
      }

      public function updated(ClanMembership $membership): void {
          // Detect left_at transition (null -> NOT NULL)
          if ($membership->wasChanged('left_at') && $membership->left_at !== null) {
              $this->dispatchSyncIfBindingComplete($membership, 'remove');
          }
          // Detect left_at reactivation (NOT NULL -> null) — re-join scenario, rare; supports it
          if ($membership->wasChanged('left_at') && $membership->left_at === null) {
              $this->dispatchSyncIfBindingComplete($membership, 'add');
          }
      }

      public function deleted(ClanMembership $membership): void {
          // Hard-delete path; treat as remove
          $this->dispatchSyncIfBindingComplete($membership, 'remove');
      }

      private function dispatchSyncIfBindingComplete(ClanMembership $membership, string $action): void {
          $membership->loadMissing(['user','clan']);
          if (($membership->user?->discord_id ?: '') === '' || ($membership->clan?->discord_role_id ?: '') === '') {
              // Skip — binding incomplete; nothing to sync
              return;
          }
          SyncDiscordRolesJob::dispatch($membership->id, $action, Auth::id());
      }
  }

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| ClanMembership lifecycle -> SyncDiscordRolesJob dispatch | Observer is the ONLY production dispatch path; Phase 2 services + Filament resources all funnel through ClanMembership writes |
| SyncDiscordRolesJob -> DiscordOutboundMessage | Job writes a single pending row; no race vs match announce because they have different message_type values |
| Horizon retry -> outbound retry | Two-layer: Horizon retries the JOB if handle() crashes; outbound table's backoff handles Discord-side delivery retries |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-06-01 | Tampering | Bot dispatches role_sync for arbitrary clans by writing directly to outbound table | mitigate | D-004 architectural — bot has only bot:write-outbound for ack endpoints, not arbitrary INSERT; this plan is web-side observer + job only |
| T-05-06-02 | EoP | Stolen Sanctum token can mark role_sync rows sent without actually delivering -> ClanMembership exists but Discord role missing | accept | v1 trade-off; admin can retry via Filament (plan 05-07); audit log captures bot causer for the mark-sent action |
| T-05-06-03 | DoS | 1000 ClanMembership creates from a seeder saturates Horizon queue + outbound table | mitigate | Pitfall 2 — Horizon worker rate-limit; outbound poll limit=20/5s; queue backpressure self-regulates |
| T-05-06-04 | Tampering | Observer fires twice for same membership state -> duplicate role_sync | mitigate | wasChanged('left_at') guard on updated(); only transitions trigger; idempotent on Discord side (Pattern 5 — Discord PUT is 204 either way) |
| T-05-06-05 | Repudiation | Job dispatched without causer attribution (CLI seeders) | accept | causerUserId is nullable; LogsActivity on DiscordOutboundMessage captures the event regardless |
| T-05-06-06 | Replay | Horizon retries handle() after partial write; outbound row exists twice | mitigate | Job handle() is idempotent-safe: writing two role_sync rows for the same membership state is harmless because the bot's PUT is idempotent (Pattern 5 + RESEARCH A5 verification — Discord returns 204 whether role was added or already present) |
| T-05-06-07 | Spoofing | Observer skip logic for empty discord_id could be bypassed if Phase 1 OAuth ever provisioned a User with empty discord_id | mitigate | Phase 1 plan 01-09 ProvisionFirstLogin requires discord_id; empty value implies seeder data and the defensive skip catches it |
