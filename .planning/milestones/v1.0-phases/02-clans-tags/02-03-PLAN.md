<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Controller `fill()` / `update()` to model | $fillable list is the ONLY arrays-from-request gate; mass-assignment attack surface |
| Filament edit form to model | Filament respects $fillable; bypasses are by-design (admin) but audited via LogsActivity |
| App code to LogsActivity | Every CRUD operation logs causer/subject; admin tampering visible in `/admin/audit` |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-02-01 | Tampering | Clan.discord_role_id mass-assignment | mitigate | NOT in `$fillable` of any My-Clan-facing route — only Filament admin can set it. Plan 02-09 MyClanProfileController explicitly filters keys (does not pass `discord_role_id`) |
| T-02-02-02 | Tampering | ClanMembership.role mass-assignment | mitigate | Role change route in plan 02-09 uses a dedicated controller action that explicitly validates the new role and checks the actor's role |
| T-02-02-03 | Information Disclosure | Player.bio HasTranslations migration | mitigate | Bio is a public-profile field — privacy gate (plan 02-05) decides whether the field appears in the DTO; the model just stores it |
| T-02-02-04 | Repudiation | All model mutations | mitigate | LogsActivity trait on every model; logFillable + logOnlyDirty; admin causer auto-captured via Spatie activitylog default |
| T-02-02-05 | Elevation of Privilege | D-009 partial unique index | mitigate | DB-level UNIQUE INDEX WHERE blocks concurrent second-active-membership attempts; covered by test in this plan |
</threat_model>
