RESEARCH.md Pattern 7 — My Clan access gate algorithm:
1. Find active membership for current user
2. If none -> render "no clan" state via Inertia::render('MyClan/Index', ['membership' => null, 'clan' => null])
3. If role in ['leader','officer'] -> render management page
4. Otherwise (member, recruit) -> redirect to /clans/{slug}

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Auth user to My Clan controllers | Must verify active Leader/Officer membership in the targeted clan, not just generic auth |
| FormRequest validation to model update | Whitelisted keys only — never `fill($request->all())` |
| Clan creation form to Leader assignment | Must be transactional — clan + membership must commit together |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-05-01 | Spoofing | Cross-clan request | mitigate | Every My Clan write route resolves the actor's active membership first, then checks `$membership->clan_id === $targetClan->id`. Route-model binding for /members/{membership} is scoped to the actor's clan via Policy::update (see ClanMembershipPolicy below). |
| T-02-05-02 | Tampering | Mass-assignment on Clan (e.g. discord_role_id, status) | mitigate | UpdateClanProfileRequest::validated() returns ONLY ['name','tag','description','country_code'] keys. The controller calls `$clan->update($request->validated())` so unknown keys cannot land. |
| T-02-05-03 | Elevation of Privilege | Promoting self to Leader / demoting other Leader | mitigate | ClanMembershipPolicy::update($actor, $membership): Officer may set role IN ['officer','member','recruit']; Leader may set any role. To transfer Leader, the existing Leader must demote themselves (validates intent). |
| T-02-05-04 | Repudiation | Audit log gap on My Clan writes | mitigate | LogsActivity trait on all models auto-logs; causer captured from auth(). Test asserts activity_log row for each mutation in MyClanManagementTest. |
| T-02-05-05 | Tampering | Reserved-slug bypass on POST /clans | mitigate | StoreClanRequest validates `name`. ClanSlugGenerator throws ReservedSlugException; controller catches and returns 422 with localized error. Validation rule also REQUIRES that the resulting slug is not in `config('clan.reserved_slugs')`. |
| T-02-05-06 | Information Disclosure | Member remove leaks user data | mitigate | "Remove" sets left_at=now() (D-009); no hard delete. PII stays in clan_memberships but `whereNull('left_at')` filters from active queries. |
| T-02-05-07 | DOS | Spam clan creation | accept | P2 has no rate limit on POST /clans; Laravel's global throttle middleware covers it. Phase 9 may add per-user limits. |
| T-02-05-08 | Audit-log tampering | LogsActivity writes outside transaction | mitigate | LogsActivity hooks Eloquent's saved event — writes happen inside the controller's DB::transaction. Verify in MyClanManagementTest via `DB::beginTransaction(); throw; assert no activity_log row`. |
</threat_model>
