Author `app/Services/NotificationDispatcher.php` per RESEARCH Pattern 2 (verbatim shape, adapted to project conventions):
- `final class NotificationDispatcher` with public method `sweepUpcoming(): void`.
- `sweepUpcoming()` calls `dispatchWindow(minutes: 60)` then `dispatchWindow(minutes: 15)`.
- `private function dispatchWindow(int $minutes): void`:
  - `$target = Carbon::now()->addMinutes($minutes);`
  - Query `GameMatch::query()` where `scheduled_at >= $target->copy()->subMinutes(3)` AND `<= $target->copy()->addMinutes(3)` AND `status = 'scheduled'` with eager-loads `['signups.user', 'hostClan.activeMemberships.user']` (eager-load required — plan 09-08 will flip strict mode).
  - For each match, merge `signups->pluck('user')` and `hostClan->activeMemberships->pluck('user')` unique by id; filter non-null and users with non-null email/discord_id.
  - For each participant: if `alreadyDispatched($user, $match, $minutes)` skip; else `$user->notify(new MatchStartingSoon($match, $minutes))`.
- `private function alreadyDispatched(User $user, GameMatch $match, int $minutes): bool`:
  - Query `DB::table('notifications')` where notifiable_type=$user->getMorphClass() AND notifiable_id=$user->id AND type='match.starting_soon' AND whereJsonContains data->match_id with $match->id AND whereJsonContains data->minutes with $minutes; return exists().

| model event→observer | Eloquent observers run in HTTP/queue worker contexts |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-04-01 | D (DoS) | Dispatcher could fire millions of notifications if scheduled_at filter misconfigured | mitigate | Idempotency check (alreadyDispatched) + ±3min boundary + sweep limited to status=scheduled; Pest test asserts boundary behavior |
| T-09-04-02 | T (Tampering) | Multi-host cron without onOneServer() races and produces dupes | mitigate | Schedule::command(...)->withoutOverlapping()->onOneServer() per Pattern 2; partial unique index on notifications data fields not added v1 (defence-in-depth A-list) |
| T-09-04-03 | I (Information Disclosure) | Match cancelled notification could leak match details to unauthorised users | mitigate | Recipient set is signups + host clan active members only; never broader; plan 09-06 bell controller scopes by auth()->user() |
| T-09-04-04 | S (Spoofing) | Observer could fire notifications for the wrong user | mitigate | Observer methods receive the model from Eloquent's event dispatcher — no user-controlled input crosses |
| T-09-04-05 | E (Elevation of Privilege) | Pruner deletes notification rows — could be abused to hide audit | accept | Notifications are NOT the audit log (activity_log is); 90-day retention is operational hygiene, not security |
