<objective>
Land the Match status state machine service and the MatchNotOpenException typed-exception class. The state machine encodes RESEARCH Pattern 4 verbatim: `draft -> open | cancelled`, `open -> locked | played | cancelled`, `locked -> played | cancelled`, `played` and `cancelled` are terminal. Every transition writes an activity_log entry with the causer and the (from, to) properties so the admin audit trail covers status flips even when triggered by side-effect services (MatchResultService in plan 04-09 flips status to 'played' atomically with the result write).

Purpose: SC-1 says signups open automatically when admin creates a Match — the wizard's `handleRecordCreation` (plan 04-09) calls MatchStatusService implicitly by setting `status='open'` in the create array, but transitions FROM `open` to `locked|played|cancelled` MUST go through this service so the audit trail captures them and invalid transitions are blocked. The MatchNotOpenException class is thin but defined here (not in plan 04-06) to avoid circular dependency — plan 04-06's MatchSignupService imports it from here.

Output: 1 service class, 1 exception class, 1 GREEN test file (replaces Wave 0 RED stub). Plan 04-09's MatchResultService consumes MatchStatusService::transition($match, 'played', $causer) atomically inside its result-write transaction.
</objective>

From RESEARCH Pattern 4 verbatim (full state machine table):
```
private const ALLOWED_TRANSITIONS = [
    'draft' => ['open', 'cancelled'],
    'open' => ['locked', 'played', 'cancelled'],
    'locked' => ['played', 'cancelled'],
    'played' => [],
    'cancelled' => [],
];
```

Transition algorithm:
1. Read $from = $match->status
2. Read $allowed = ALLOWED_TRANSITIONS[$from] ?? []
3. If $to NOT in $allowed, throw DomainException(__('matches.status.error.invalid_transition', ['from' => $from, 'to' => $to]))
4. Wrap in DB::transaction: $match->update(['status' => $to]); then activity()->causedBy($causer)->performedOn($match)->withProperties(['from' => $from, 'to' => $to])->log("Match status transition")

NOTE on the `Match` class FQN at use sites containing `match($x)` expressions: per Pitfall 5 alias on import. In MatchStatusService specifically, no `match($x)` expressions appear, so plain `use App\Models\Match;` is acceptable. If a reviewer is uncomfortable, switch to `use App\Models\Match as MatchModel;` for safety.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Caller to transition() | Service is stateless; caller MUST pass User $causer (no Auth::user() inside the service — testability + reuse for system-driven transitions via cron in Phase 5+) |
| Service to activity_log | Service writes activity log via causer-bound activity()->log() call; admin tampering with status is auditable |
| Service to DB | DB::transaction wraps both the status update AND the activity log write — partial state (status updated, log missing) is impossible |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-04-01 | Tampering | Invalid status transition via service bypass | mitigate | DB-layer matches_status_check CHECK (plan 04-02) blocks any status string not in the enum even if a future caller bypasses the service; service-layer ALLOWED_TRANSITIONS blocks valid-string-but-invalid-transition writes (e.g., played -> open) |
| T-04-04-02 | Repudiation | Status flip with no audit trail | mitigate | activity()->causedBy($causer)->withProperties(['from','to'])->log() writes a typed audit row inside the transaction; partial state impossible |
| T-04-04-03 | Tampering | Filament admin manually edits Match.status field | mitigate | Filament MatchResource (plan 04-09) uses ->disabled() on the status field on Edit page; transitions happen via explicit Filament Actions that route through MatchStatusService (Pitfall 7) |
| T-04-04-04 | Information Disclosure | Audit log leaks internal status strings | accept | Status values are part of the public Match contract (rendered on /matches calendar via MatchStatusBadge); not sensitive |
</threat_model>
