From RESEARCH Pattern 3 (verbatim — full method body):
```php
public function materialise(MatchModel $match): int
{
    return DB::transaction(function () use ($match): int {
        $match->loadMissing('gameMatchType.roleLimits');
        $count = 0;

        foreach ($match->gameMatchType->roleLimits as $limit) {
            for ($i = 0; $i < $limit->capacity; $i++) {
                MatchSlot::create([
                    'match_id' => $match->id,
                    'game_role_id' => $limit->game_role_id,
                    'slot_index' => $i,
                    'sort_order' => $limit->sort_order,
                ]);
                $count++;
            }
        }

        return $count;
    });
}
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Caller (Filament wizard or future officer-Create controller) to materialise() | Service assumes $match is freshly created with no pre-existing slots; idempotency is enforced by DB unique constraint, not by the service |
| Service to MatchSlot writes | All writes inside DB::transaction — partial materialisation (e.g., 30 of 50 slots inserted before FK error) is impossible |
| Service to GameMatchType.roleLimits read | loadMissing reads inside the transaction; concurrent admin RoleLimit edits during materialise are blocked by the read's snapshot semantics (Postgres READ COMMITTED) |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
