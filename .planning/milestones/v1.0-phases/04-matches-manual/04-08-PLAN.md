The observer-managed events table makes the calendar query O(1) on a single indexed table.

RESEARCH Pattern 8 (verbatim — full observer body):

```
<?php
declare(strict_types=1);

namespace App\Observers;

use App\Models\Event;
use App\Models\Match as MatchModel;

class MatchObserver
{
    public function saved(MatchModel $match): void
    {
        $shouldHaveEvent = $match->is_public && $match->status !== 'cancelled';

        if ($shouldHaveEvent) {
            Event::updateOrCreate(
                ['eventable_type' => MatchModel::class, 'eventable_id' => $match->id],
                [
                    'starts_at' => $match->scheduled_at,
                    'ends_at' => null,
                    'title' => $match->getTranslations('title'),
                    'is_public' => $match->is_public,
                ],
            );
        } else {
            Event::where('eventable_type', MatchModel::class)
                ->where('eventable_id', $match->id)
                ->delete();
        }
    }

    public function deleted(MatchModel $match): void
    {
        Event::where('eventable_type', MatchModel::class)
            ->where('eventable_id', $match->id)
            ->delete();
    }
}
```

NOTE on Pitfall 12 (mass updates skip observers): the test must NOT use Model::query()->update() — that skips observers. Use $match->update() or $match->save() which fire the saved event.

NOTE on the `Match` class import: in MatchObserver.php, use `use App\Models\Match as MatchModel;` (Pitfall 5). The `eventable_type` value stored in the events table is `App\Models\Match` (the FQN string Laravel writes via morph map fallback). When comparing in tests, `Event::where('eventable_type', Match::class)` works because `Match::class` resolves to the FQN string at PHP runtime.

NOTE on the events.title column: it's a denormalised CACHE of Match.title (Pattern 8 + Assumption A3). The observer's saved() listener overwrites it every time the Match saves — so admin edits to Match.title propagate automatically to the events row. This is what makes the /matches calendar query able to read events.title directly without joining matches.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Match save event to observer | Observer runs in the same DB::transaction as the model save (Eloquent default) — observer write rolls back if outer transaction fails |
| Observer to events table | events_one_per_owner UNIQUE (plan 04-02) is the integrity gate; updateOrCreate keeps the observer idempotent |
| Bulk update bypass | Model::query()->update() skips model events; mitigated by Filament conventions (single-record actions) and Pitfall 12 documentation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-08-01 | Tampering | Cancelled match retains Event row (calendar drift) | mitigate | saved() listener deletes Event when status=cancelled OR is_public=false; covered by 2 Pest tests in this plan |
| T-04-08-02 | Tampering | events_one_per_owner UNIQUE violated by observer | mitigate | updateOrCreate's first arg matches the UNIQUE columns; second arg is the upsert payload; Laravel's updateOrCreate is idempotent against this constraint |
| T-04-08-03 | Tampering | events.title cache drift from Match.title edit | mitigate | saved() listener overwrites events.title on every save; Pitfall 12 (bulk update) is the only escape vector |
| T-04-08-04 | Repudiation | Observer writes lack causer attribution | accept | Event model has LogsActivity (plan 04-03) which captures the implicit causer via Spatie's default auth resolver; observer-driven writes attribute to the user who triggered the save (admin or organiser) |
| T-04-08-05 | Information Disclosure | Polymorphic eventable_type spoofing | mitigate | eventable_type is set BY the observer only (Match::class); never user input. Filament EventResource (plan 04-09 EventResource) is read-only |
| T-04-08-06 | Tampering | Observer execution order with multiple listeners | accept | Phase 4 has only MatchObserver on Match; Phase 6 may add TournamentObserver on Tournament — separate class, no interference |
