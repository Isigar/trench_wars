<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → GET /clans/{slug} (public) | Viewer-state props are computed server-side; the client cannot fabricate eligibility |
| Browser → POST /clans/{slug}/apply | Submission re-validated server-side by the service (plan 10-02/10-03); the UI guard is convenience-only |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-06-01 | Spoofing | client showing the Apply block when ineligible | mitigate | The `showApplyBlock` guard is UX only; the server-side service guards (plan 10-02) and the web controller (plan 10-03) are authoritative — a forged POST still hits all three eligibility checks |
| T-10-06-02 | Information disclosure | leaking other users' application state | mitigate | viewerHasPendingApplication is scoped to `applicant_user_id = $viewer->id` — never reveals others' applications |
| T-10-06-03 | Cross-site scripting | cover message rendered later | accept | This plan only submits the message (stored as plain text); rendering on the review side already uses text binding (no v-html) per Phase 2 |
| T-10-06-SC | Tampering | npm installs | accept | No package installs — reuses existing Inertia/Vue deps |
</threat_model>
