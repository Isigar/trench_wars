<objective>
Build both CLAN-04 recruiting-toggle surfaces: the clan leader/officer self-service toggle (MyClan settings) and the admin override (Filament ClanResource).

The toggle flows through the existing `UpdateClanProfileRequest` → `MyClanProfileController::update` → `$clan->update($request->validated())` pipeline, which is already authorized by `ClanPolicy::update` (active leader/officer). Adding `accepts_applications` to the request's `rules()` whitelists it; the MyClan profile form gains a checkbox bound to it; `ClanData` exposes the current value so the form reflects state. Filament admins get a `Toggle` in the clan form.

Purpose: leaders control whether their clan accepts applications (the front half of CLAN-04; the rejection half is enforced by the service guard from plan 10-02).
Output: DTO field + shared-types regen, request rule, MyClan toggle UI, Filament toggle, admin i18n, feature test.
</objective>

```php
public function authorize(): bool {
    $clan = $this->route('clan');
    return $this->user()?->can('update', $clan) ?? false;
}
public function rules(): array {
    return [
        'name' => ['sometimes', 'required', 'string', 'max:255'],
        'tag' => ['sometimes', 'required', 'string', "min:{$tagMin}", "max:{$tagMax}", 'regex:/^[A-Za-z0-9_-]+$/'],
        'description' => ['sometimes', 'nullable', 'string', "max:{$descMax}"],
        'country_code' => ['sometimes', 'nullable', 'string', 'size:2'],
    ];
}
```
