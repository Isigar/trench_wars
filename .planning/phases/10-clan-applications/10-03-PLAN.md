<objective>
Wire the two submission controllers and their routes + the PHP test surface.

The bot endpoint `POST /api/bot/clans/{clan:slug}/applications` (CLAN-02) lives under the acts-as-user route group — `auth()->user()` is the rebound human, never the bot service account (T-05-04-01 mitigation already enforced by the `bot.acts-as` middleware). Purpose: expose the service's apply() to both surfaces with the project's canonical error-mapping conventions.
Output: 2 controllers, 2 route edits, 3 test files.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Discord bot → /api/bot (Sanctum + abilities + bot.acts-as) | Bot presents a token + X-Bot-Acts-As-User; middleware rebinds auth to the human |
| Browser → /clans/{slug}/apply (web session) | Authenticated session user is the applicant |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-03-01 | Spoofing | X-Bot-Acts-As-User on the bot endpoint | mitigate | Route is under `abilities:bot:act-as-user` + `bot.acts-as`; `Auth::user()` is the rebound human, never the bot service account (T-05-04-01). The AbilitiesTest proves 403 without the act-as ability |
| T-10-03-02 | Elevation of Privilege | applying as another user | mitigate | Web controller uses `$request->user()` (session); bot controller uses the acts-as-rebound user — neither accepts a client/bot-supplied applicant id |
| T-10-03-03 | Tampering | bypassing eligibility in the controller | mitigate | Both controllers delegate every write to ClanApplicationService::apply() (D-004); the three guards live in the service, not the controllers |
| T-10-03-04 | Information disclosure | error messages leaking internals | accept | Errors are i18n-keyed strings (`bot.errors.*` / `clans.applications.error.*`), not stack traces or DB messages |
| T-10-03-SC | Tampering | composer installs | accept | No package installs in this plan |
</threat_model>
