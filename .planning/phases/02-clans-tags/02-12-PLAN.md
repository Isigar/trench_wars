<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Admin user to Filament resource | admin-access permission gate (Phase 1 plan 11/12) |
| Form data to model save | Filament respects $fillable; admin can bypass My Clan FormRequest guards (by design) |
| RelationManager actions | Admin can directly manipulate ClanMembership rows incl. set role to 'leader' |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-09-01 | Elevation of Privilege | Non-admin access to Filament | mitigate | Phase-1 AdminPanelProvider gates entire /admin under `admin-access` permission via User::canAccessPanel (verified by FilamentPanelAccessTest) |
| T-02-09-02 | Tampering | Admin sets discord_role_id | accept | Admin's job. Audit captures change. |
| T-02-09-03 | Repudiation | Admin actions unaudited | mitigate | LogsActivity on Clan + ClanMembership models. Audit tab in ClanResource shows per-record history. |
| T-02-09-04 | Tampering | KeyValue submits null instead of {} | mitigate | `->default(['en' => ''])` + mutateFormDataBeforeSave coerces null to ['en' => ''] (Pitfall 6) |
| T-02-09-05 | Tampering | discord_guild row deletion via admin | accept | DiscordGuildResource ships in plan 02-13 with no-Delete; this plan's ClanResource does not touch discord_guild |
</threat_model>
