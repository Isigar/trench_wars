<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Leader/Officer to application accept | Must verify acceptor is in the target clan |
| Applicant to application cancel | Must verify acting user is applicant_user_id |
| Application accept to membership creation | Must be atomic |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-07-01 | Spoofing | Accept-application as wrong actor | mitigate | Controller resolves actor's active Leader/Officer membership; verifies clan match |
| T-02-07-02 | Tampering | Re-accept already-accepted application | mitigate | Service checks status === 'pending' |
| T-02-07-03 | Tampering | Accept application for applicant who joined another clan | mitigate | Service rechecks "applicant has no active membership" inside the DB transaction; D-009 partial index is the last-line defence |
| T-02-07-04 | Information Disclosure | Cross-clan application listing | mitigate | MyClanController scopes applications via membership->clan->applications relation; controller never returns all-clans applications |
| T-02-07-05 | XSS | Application message rendering | mitigate | Vue auto-escapes message; no v-html |
| T-02-07-06 | Repudiation | Decline reason omitted from audit | accept | Status change is logged via LogsActivity with default description; decline reason text is optional and stored in `message` only at apply time. P2 has no decline-reason form. |
</threat_model>
