# Railway Deploy Walkthrough — Phase 1

> Source: D-014 LOCKED + 01-RESEARCH.md Pitfall 6.
> One Railway project. 5 services (web, worker, bot, rcon-worker, plus the two plugins).

---

## Prerequisites

1. A Railway account (https://railway.com) with the project owner role.
2. Railway CLI installed locally — `npm install -g @railway/cli` then `railway login`.
3. Discord OAuth credentials (CLIENT_ID + CLIENT_SECRET) from https://discord.com/developers/applications. The production redirect URI registered there must EXACTLY match the production `DISCORD_REDIRECT_URI` env value (Pitfall 2 from `01-RESEARCH.md`).
4. The repo cloned locally at `trenchwars/` with `apps/web/`, `apps/bot/`, `apps/rcon-worker/`, `packages/shared-types/`.

---

## Step 1 — Create the Railway project

```bash
cd trenchwars
railway init
```

Pick "Empty Project" and name it `trenchwars-prod`. Railway generates a `RAILWAY_PROJECT_ID` and links the local directory.

---

## Step 2 — Attach Postgres + Redis plugins

In the Railway dashboard:
1. **+ New** → **Database** → **Add PostgreSQL**. Pick PostgreSQL 16. Railway provisions a managed instance and exposes `DATABASE_URL` (and individual `PGHOST`, `PGPORT`, `PGUSER`, `PGPASSWORD`, `PGDATABASE`) as service env.
2. **+ New** → **Database** → **Add Redis**. Pick Redis 7. Exposes `REDIS_URL`.

Both plugins live inside the same project; the variables become referenceable across services via `${{Postgres.DATABASE_URL}}` and `${{Redis.REDIS_URL}}`.

---

## Step 3 — Create env groups

In the Railway dashboard, navigate to **Project Settings → Variables → Shared Variables**. Create FOUR shared variable groups (logical, not enforced):

| Group | Variables |
|---|---|
| `app` | `APP_ENV=production`, `APP_DEBUG=false`, `APP_KEY=<run \`php artisan key:generate --show\` locally and paste>`, `APP_URL=https://your-trenchwars-domain.com`, `LOG_CHANNEL=stderr`, `LOG_DEPRECATIONS_CHANNEL=null` |
| `database` | `DB_CONNECTION=pgsql`, `DB_URL=${{Postgres.DATABASE_URL}}` (Railway-substituted) |
| `redis` | `REDIS_URL=${{Redis.REDIS_URL}}`, `SESSION_DRIVER=redis`, `CACHE_STORE=redis`, `QUEUE_CONNECTION=redis` |
| `discord` | `DISCORD_CLIENT_ID=<value>`, `DISCORD_CLIENT_SECRET=<value>`, `DISCORD_REDIRECT_URI=https://your-trenchwars-domain.com/auth/discord/callback` |

Add `app` + `database` + `redis` + `discord` to the **web** and **worker** services.
Add `app` + `discord` to the **bot** service.
Add `app` + `redis` to the **rcon-worker** service. (RCON-specific keys come in Phase 8.)

---

## Step 4 — Configure four services

For each service, **+ New** → **Empty Service**, name it, then attach to the same GitHub repo (or use `railway up` to deploy from local).

### `web` service

1. **Settings → Service → Source → Repo**: trenchwars repo.
2. **Settings → Service → Source → Root Directory**: `apps/web`.
3. **Settings → Variables → Add**: `NIXPACKS_PHP_ROOT_DIR=/app/public` (Pitfall 6).
4. **Settings → Networking → Public Networking**: enable; Railway issues `https://web-production-xxxx.up.railway.app`.
5. **Settings → Health Check** → path: `/up`, timeout: 30s.
6. Variables: attach groups `app`, `database`, `redis`, `discord`.

### `worker` service

1. **+ New** → **Empty Service** → **Source → Repo**: trenchwars (same repo).
2. **Root Directory**: `apps/web` (same as web).
3. **Settings → Deploy → Custom Start Command**: `php artisan queue:work --tries=3 --backoff=10 --timeout=120`.
4. Variables: attach groups `app`, `database`, `redis`, `discord` (same as web).
5. NO public networking (worker is an internal job runner).

### `bot` service

1. **Root Directory**: `apps/bot`.
2. **Settings → Variables → Add**: `NIXPACKS_NODE_VERSION=22`.
3. Variables: attach groups `app`, `discord`.
4. NO public networking.

### `rcon-worker` service

1. **Root Directory**: `apps/rcon-worker`.
2. **NIXPACKS_NODE_VERSION=22**.
3. Variables: attach groups `app`, `redis`.
4. NO public networking.

---

## Step 5 — Push to deploy

```bash
git push origin main
```

Railway auto-builds each service whose path filter matches the changed files. Tail the logs:

```bash
railway logs --service web
railway logs --service bot
```

---

## Step 6 — Smoke check

1. Open the public web URL in a browser. Landing page renders with Discord CTA.
2. Click "Log in with Discord". Complete OAuth on Discord. Lands on `/` with success flash.
3. Run the admin grant command via Railway CLI:
   ```bash
   railway run --service web php artisan trenchwars:make-admin <YOUR_DISCORD_USER_ID>
   ```
4. Open `/admin`. The Filament panel should render with the trench-military theme; you should see User, Player, Role, Permission resources + the Audit page.

## Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| Build error: "no PHP detected" | Root Directory not set on the web service | Settings → Source → Root Directory = `apps/web` |
| "Couldn't reach Discord" toast | Discord redirect URI mismatch | Discord Developer Portal: register the EXACT production URL with no trailing slash |
| 419 on form POST | `<meta csrf-token>` re-introduced into Blade | `git diff` `apps/web/resources/views/app.blade.php` — remove any csrf-token meta (Pitfall 3) |
| Filament panel CSS broken | Filament theme bundle missing | Confirm `pnpm run build` ran during Nixpacks build phase; check `apps/web/public/build/filament/manifest.json` exists |
| 403 on /admin even after make-admin | Spatie permission guard mismatch (Pitfall 4) | Verify `config/permission.php` `default_guard_name => 'web'` is set |
| Worker service idle | queue:work not running | Settings → Deploy → Custom Start Command on the worker service set to `php artisan queue:work --tries=3` |
