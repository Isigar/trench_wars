Canonical second-tier resource pattern (verified `apps/web/app/Filament/Resources/ClanMembershipResource.php`):
- Standalone Filament Resource for a child entity that also appears as a RelationManager on the parent Resource
- This is Filament v3's idiomatic answer to nested RelationManagers (Pattern 2)
- Phase 2 used it for Clan → ClanMembership; Phase 3 uses it for Game → GameMatchType

From RESEARCH.md Pattern 3 (scoped Select inside RelationManager — verbatim verified by Context7):
```
Forms\Components\Select::make('game_role_id')
    ->label(__('admin.game_match_type_role_limit.fields.role'))
    ->options(function (RelationManager $livewire): array {
        /** @var \App\Models\GameMatchType $matchType */
        $matchType = $livewire->getOwnerRecord();
        return $matchType->game->roles()
            ->orderBy('sort_order')
            ->get()
            ->mapWithKeys(fn ($role) => [$role->id => $role->display_name['en'] ?? $role->key])
            ->toArray();
    })
    ->required()
    ->searchable(),
```

The `getOwnerRecord()` method on RelationManager returns the parent record (the GameMatchType being edited). The Select's options are reactive — Filament re-evaluates the lambda each render, so newly-added Roles to the parent Game appear in the dropdown immediately.

From plan 03-06 (verified plan file): MatchTypesRelationManager's EditAction currently uses the default modal-based edit. Pattern 2 click-through requires override:
```
Tables\Actions\EditAction::make()
    ->url(fn (\App\Models\GameMatchType $record) => \App\Filament\Resources\GameMatchTypeResource::getUrl('edit', ['record' => $record]))
```
This is the canonical Filament v3 navigation idiom — the lambda returns a fully-qualified URL string; Filament wires the action button to a native browser navigation instead of opening a modal.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Admin UI to RoleLimit row writes | Filament Select scoping (Pattern 3) prevents cross-game role picks via UI |
| RoleLimit row creation across API / Console | Model saving() listener (plan 03-03) catches non-UI writes |
| Capacity input | Filament `->numeric()->minValue(0)` is the form-layer gate; DB CHECK (plan 03-02) is the second gate |
| MatchTypesRelationManager EditAction click | URL override navigates to standalone GameMatchTypeResource; the click-through is the Pattern 2 contract |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-07-01 | Tampering | Cross-game RoleLimit creation via admin UI (Pitfall 10 admin half) | mitigate | RoleLimitsRelationManager Select scoped via getOwnerRecord()->game->roles() (Pattern 3); admin cannot select a role from a different game |
| T-03-07-02 | Tampering | Negative capacity via admin UI | mitigate | Forms\Components\TextInput::make('capacity')->numeric()->minValue(0); DB CHECK is second prong |
| T-03-07-03 | Elevation of Privilege | Non-admin reaches /admin/game-match-types | mitigate | Inherited Phase 1 admin-access gate |
| T-03-07-04 | Tampering | KeyValue submits null for name / description | mitigate | mutateFormDataBefore* coerces both translatable JSONB fields (Pitfall 2 with TWO fields per Pattern 2 + Pitfall 2 note) |
| T-03-07-05 | Repudiation | RoleLimit edits unaudited | mitigate | LogsActivity on GameMatchTypeRoleLimit (plan 03-03); per-resource Audit tab inherits |
| T-03-07-06 | Spoofing | Admin creates standalone GameMatchType without selecting Game | mitigate | CreateGameMatchType form requires `game_id` Select; field is required |
</threat_model>
