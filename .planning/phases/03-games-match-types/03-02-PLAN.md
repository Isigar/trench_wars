Land the 4 Phase 3 migrations in the canonical Phase 1/2 raw-SQL idiom: UUID PKs with `gen_random_uuid()` default, `timestamptz` upgrade on `created_at`/`updated_at`, key-format CHECK constraints (`key ~ '^[a-z0-9_]+$'`) on every `key` column, capacity CHECK (`capacity >= 0`) on `game_match_type_role_limits`, explicit composite UNIQUE index names per Pitfall 1, cascade-on-delete FK chain per Pitfall 7, down migrations drop indexes BEFORE tables.

Purpose: Establish the relational backbone for D-007. Every Phase 3 model (plan 03-03), seeder (plan 03-05), and Filament resource (plan 03-06, 03-07) compiles against these tables. The composite `(game_id, key)` UNIQUE is what makes the seeder's `firstOrCreate` idempotent (Pattern 5) and what enforces SC-1 ("`(game_id, key)` uniqueness enforced"). The capacity CHECK is the second prong of input validation (Pattern V5 ASVS L1) — Filament form `->minValue(0)` is the first prong; the DB CHECK survives every future API or console write.
