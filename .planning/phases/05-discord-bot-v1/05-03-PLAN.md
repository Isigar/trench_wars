<objective>
Wave 2: ship the auth substrate for every `/api/bot/*` route — `ResolveBotActsAsUser` middleware (RESEARCH Pattern 1) + Sanctum CheckAbilities alias registration in `bootstrap/app.php`. This is the SC-5 mechanical guarantee that activity_log entries attribute the HUMAN behind each Discord action (via `X-Bot-Acts-As-User: <discord_id>` resolved to a User row and rebound via `Auth::onceUsingId`), not the bot's Sanctum-token-owning service account.

The middleware is intentionally tolerant of missing header on read-only routes (Pitfall 7) — it ONLY applies on the routes explicitly listed in plan 05-04's `/api/bot/*` route group with `abilities:bot:act-as-user`. The route grouping (in plan 05-04) decides which endpoints require acts-as, NOT a global gate on the whole group.

This plan also lands two of the 11 Wave 0 RED stubs as GREEN: ResolveBotActsAsUserMiddlewareTest (unit-level middleware behaviour) + BotApiAuthMatrixTest (Sanctum auth matrix — token presence + ability check + acts-as resolution combined). The matrix test is the SC-5 verification surface; controller-level tests in plan 05-04 build on top.

Purpose: SC-5 — "all bot->web traffic uses Sanctum scoped token + X-Bot-Acts-As-User header; audit log entries correctly attribute the human causer behind every Discord action."

Output: 1 middleware + bootstrap/app.php alias edit + sanctum.php confirmation + 2 GREEN test files.

ResolveBotActsAsUser middleware (verbatim from RESEARCH Pattern 1):

  namespace App\Http\Middleware
  final class ResolveBotActsAsUser
  public function handle(Request $request, Closure $next): Response

The handle() body (verbatim from RESEARCH):

  - read header X-Bot-Acts-As-User
  - if null: return $next($request) (Pitfall 7 — read-only routes tolerate missing header)
  - validate snowflake shape via Str::isUuid is wrong here; use ctype_digit + length 17..20 (Discord snowflake constraint)
  - lookup User::where('discord_id', $actsAs)->first()
  - if null: return response()->json(['error' => 'acts_as_user_unknown', 'message' => __('bot.errors.acts_as_unknown')], 422)
  - Auth::onceUsingId($user->id) — rebinds request-scope auth
  - return $next($request)

NOTE on Auth::onceUsingId vs Auth::loginUsingId — onceUsingId does NOT persist a session; perfect for stateless API auth. Sanctum bearer auth is already stateless on /api/bot/*; onceUsingId stacks on top without polluting session storage.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Bot Sanctum token -> /api/bot/* routes | Token leakage = full Discord-side compromise (Pitfall 3); abilities are the defence-in-depth |
| Bearer token -> token abilities -> acts-as user resolution | 3-layer gate; each layer is independent |
| X-Bot-Acts-As-User header -> User lookup | Bot can claim to act as ANY user with valid discord_id (Spoofing); mitigated only by token rotation + last_used_at monitoring (Phase 5 has no anomaly detection — accept) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-03-01 | Spoofing / EoP | Stolen bot token used to spoof admin via X-Bot-Acts-As-User | accept | Sanctum token rotation (90-day expires_at); plan 05-07 Artisan revoke command; audit-log forensically attributes to spoofed user; no IDS in v1 |
| T-05-03-02 | EoP | Token lacks bot:act-as-user ability but request includes X-Bot-Acts-As-User | mitigate | abilities middleware runs BEFORE bot.acts-as in route stack; abilities returns 403 first |
| T-05-03-03 | Tampering | Expired token still accepted | mitigate | Sanctum checks expires_at on every request; expired = 401 (BotApiAuthMatrixTest covers) |
| T-05-03-04 | Spoofing | Unknown discord_id header value | mitigate | Middleware returns 422 with bot.errors.acts_as_unknown; never silently passes through |
| T-05-03-05 | Information Disclosure | Stack trace returned on malformed header | mitigate | Middleware catches non-numeric / overflow header values and returns generic 422 (test covers) |
| T-05-03-06 | Repudiation | Auth::onceUsingId persists session and leaks across requests | mitigate | onceUsingId by definition does NOT persist; documented test asserts session NOT touched |
| T-05-03-07 | Tampering | bot.acts-as middleware globally applied to /api/bot/* breaking outbound ack flow (Pitfall 7) | mitigate | This plan does NOT apply the middleware globally; plan 05-04 applies it only on the acts-as-required subset |
</threat_model>
